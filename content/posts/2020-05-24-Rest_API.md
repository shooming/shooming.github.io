## REST를 구성하는 스타일
⁃ client-server
⁃ stateless
⁃ cache
⁃ uniform interface *
⁃ layered system
⁃ code-on-demand(optional) 자바스크립트

## Unifrom inteface의 제약조건
⁃ 리소스가 uri로 식별되면된다.
⁃ representations전송을 통해 리소스를 조작해야한다.(리소스를 만들거나, 업뎃, 삭제시 http 메세지에 표현을 담아서 전달해야함)
⁃ self-descriptive messages(메시지는 스스로를 설명해야한다.)
⁃ hypermedia as the engine of application state(HATEOAS) 애플리케이션의 상태는 hyperlink를 이용해 전이되어야한다.

## Unifrom Interface를 사용하는 이유

### 독립적 진화
⁃ 서버와 클라이언트가 각각 독립적으로 진화한다.
⁃ 서버의 기능이 변경된어도 클라이언트를 업데이트할 필요가 없다.
⁃ Rest를 만들게된 계기

### 웹
⁃ 웹. 페이지 변경했다고 웹 브라우저 업뎃 필요없음
⁃ 웹브라우저 업데이트 했다고 웹 페이지 변경할 필요없음
⁃ HTTP 명세가 변경되어도 웹은 잘 동작한다.
⁃ HTML 명세가 변경되어도 웹은 잘 동작한다.

### REST가 웹에 진화에 준 도움
⁃ HTTP에 지속적으로 영향을 줌
⁃ Host 헤더 추가
⁃ 길이 제한을 다루는방법이 명시(414 URI Too Long 등)
⁃ URI에서 리소스의 정의가 추상적으로 변경
⁃ HTTP/1.1 명세 최신판에서 REST에 대한 언급이 들어감

### json의 Self-descriptive 충족시키는 법
1. Media Type
    1. 미디어 타입을 하나 정의한다.
    2. 미디어 타입 문서를 작성하여 json의 key값의 의미를  정의함
    3. IANA에 미디어 타입을 등록함. 이 때 만든 문서를 미디어 타입의 명세로 등록함.

2.Profile
1. 링크에 realtion을 proxy 하면 정보가 담긴 문서 링크걸 수 있음
2. 메시지를 보는 사람이 명세를 찾아 갈 수있음

단점이 있음
1. 클라이언트가 Link 헤더(RFC5988)와 profile(RFC 6906)을 이해해야함
2. Content negoitiation을 할 수 없음
